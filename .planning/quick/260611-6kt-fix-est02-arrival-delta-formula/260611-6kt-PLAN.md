<objective>
Fix the EST-02 travel-time estimate correctness bug in the `estimate_travel_time` endpoint:
`osrm_delta` currently compares a per-stop cumulative travel time against a single
grand-total scalar (`baseline_total` = sum of ALL original legs). This is meaningless â€”
with zero edits (`proposed_stops == original_stops`) every non-final stop shows a large
spurious delta.

Replace the scalar with a per-stop `baseline_cumulative` array and compare each proposed
stop's cumulative time against the SAME stop's cumulative time on the original route
(per-stop vs per-stop). New stops propagate the delta from the nearest preceding existing
stop. Update frontend tooltip/info wording to match the corrected "vs the same stop on the
original route" semantics.

Purpose: Estimates become trustworthy â€” no edits means zero delta, edits show the real
arrival shift at each shared stop.
Output: Corrected backend formula + updated/new regression tests + reworded frontend strings.

Constraints: No new dependencies. No API contract changes (Pydantic models / TS types
unchanged â€” only internal computation and string literals). Single-worker, in-memory only.
</objective>
