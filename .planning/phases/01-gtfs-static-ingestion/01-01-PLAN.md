<objective>
Deliver the Walking Skeleton for the GTFS milestone: a backend that downloads MTD's GTFS static feed on startup, parses it with gtfs-kit, holds it in memory on `app.state.gtfs_feed`, refreshes it in the background, guards in-flight requests with HTTP 503, and exposes a `/api/gtfs/status` endpoint reachable from the frontend through the existing Next.js BFF proxy. The Docker image must build with the GDAL system dependency that gtfs-kit requires.

This is the thinnest end-to-end vertical slice that proves the entire ingestion stack works: Docker builds (GDAL) -> feed loads on startup -> a real endpoint returns real feed data -> background refresh keeps it fresh -> 503 guard protects callers before load completes. Every downstream phase (Export, Trip Updates, TripModifications, Estimation) depends on `app.state.gtfs_feed` existing, so this plan establishes that contract.

Purpose: De-risk the GDAL/Docker build and establish the in-memory GTFS feed singleton that all later phases consume.
Output: `backend/app/services/gtfs.py`, `backend/app/routers/gtfs.py`, extended `config.py`, `main.py`, `requirements.txt`, `Dockerfile`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MTD GTFS server -> backend | The backend downloads an untrusted ~26 MB zip from `https://mtd.dev/gtfs.zip` over the network and parses it in-process |
| frontend (browser) -> Next.js BFF -> backend | Browser reaches `/api/gtfs/status`; only the BFF talks to FastAPI (no direct browser->FastAPI) |
| pip/apt install (build time) -> image | New `gtfs-kit` package and `libgdal-dev` apt package introduced into the build |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Denial of Service | `load_gtfs_feed` download | mitigate | 60s httpx timeout on the download; parse runs in an executor so a slow/large feed never blocks the event loop or other requests |
| T-01-02 | Denial of Service | startup feed load | mitigate | 503 guard returns immediately while loading; a failed load leaves `app.state.gtfs_feed = None` and the service stays up (warn-don't-crash per D-04) |
| T-01-03 | Tampering | malicious/corrupt GTFS zip | mitigate | Parse failures are caught in load/refresh paths and logged as warnings; the prior good feed is retained on refresh failure, feed is never partially applied (assigned only on success) |
| T-01-04 | Information Disclosure | `/api/gtfs/status` response | accept | Status returns only aggregate counts and a timestamp — no PII, no secrets; feed data is public MTD schedule data |
| T-01-05 | Denial of Service | background refresh loop | mitigate | Interval is bounded by `gtfs_refresh_interval_hours` (default 6h); the task is cancelled cleanly on shutdown to avoid orphaned loops; failures continue the loop without tight-looping (sleep precedes the fetch) |
| T-01-SC | Tampering | pip/apt installs (`gtfs-kit`, `libgdal-dev`) | mitigate | `gtfs-kit` is an established PyPI package (RESEARCH-selected over partridge/pygtfs); `libgdal-dev` is a standard Debian package. See Package Legitimacy note below. |
</threat_model>
